很多人对 vibe coding 有一个误解：以为它就是"给 AI 一句话，然后等它自动写完代码"。
但真正有效的 vibe coding，前提恰恰是：人尽量不要直接看代码，也不要频繁介入代码细节。
它的目标很简单：
尽可能减少人工干预；
尽可能提高最终交付质量；
至少避免在线上环境出现恶性 bug。换句话说，vibe coding 不是"不负责任地让 AI 写代码"，而是靠一套流程，把 AI 的产出变得可控、可审查、可验证。下面是我认为更合理的一套流程。
1. 先做上下文工程
需求来了之后，不要急着让 AI 开始写代码。
真正开始编码之前，第一步应该是：给 AI 提供足够完整的上下文信息。 上下文质量越高，后续写出来的 PRD、实现方案和代码质量就越稳定。
使用第一梯队模型
至少应该用能力足够强的模型：GPT-5.4、Claude Opus 4.7，或同级别模型。
原因很简单：上下文理解、架构判断、需求拆解和风险识别，都是高阶能力。如果一开始模型理解错了，后面所有步骤都会被带偏。
在项目目录中完成上下文分析
如果是现有项目的新需求，一定要在项目目录里做这件事。
这样 AI 才能读取和理解：当前项目结构、现有代码风格、数据库设计、API 设计、业务逻辑、测试方式，以及构建和部署流程。
这一步的目标不是写代码，而是让 AI 先"看懂项目"。
2. 准备一个 Review Skill
接下来，每一个关键步骤都应该由两个 AI 配合完成。
分成两个角色：
Writer：负责生成内容；
Reviewer：负责审查、质疑和提出修改意见。Claude Code、Codex 这类 AI Coding 工具，通常支持调用 subagent 或外部工具。可以通过写一个 skill，让 Claude Code 调用 Codex，或者让 Codex 调用 Claude Code，实现一种类似"AI 结对编程"的流程。
为什么需要两个 AI？
单个 AI 很容易自洽。它写出来的东西，看上去逻辑完整，但可能隐藏了很多问题：需求理解有偏差、技术方案过度复杂、边界条件遗漏、数据结构设计不合理、测试覆盖不足、没有考虑线上兼容性。
引入 Reviewer 的价值就在于：让另一个 AI 专门找问题。
Writer 负责推进，Reviewer 负责挑刺。两者反复迭代，质量会明显提高。
3. 写 PRD
第一步正式产出，应该是 PRD，而不是代码。
先给 AI 一个粗旷的需求说明，然后和 AI 结对讨论，把需求逐步补全，最后生成一份尽可能完整的 PRD。
PRD 应该解决什么问题？
一份合格的 PRD，至少应该说清楚：
这个需求要解决什么问题；
用户是谁；
核心使用场景是什么；
具体功能范围是什么；
哪些内容不在本次范围内；
有哪些边界情况；
成功标准是什么；
是否涉及权限、计费、通知、数据迁移等复杂逻辑。
用 Writer + Reviewer 反复打磨
PRD 阶段同样需要两个 AI 协作。流程是：Writer 写出第一版 PRD → Reviewer 检查问题 → Writer 修改 → Reviewer 再次检查 → 重复，直到双方基本达成共识。
这一步不要怕慢。PRD 写得越清楚，后面写代码越快，返工越少。
4. 写实现文档
PRD 确定之后，不要马上进入代码实现。第二步应该是：根据 PRD 写一份详细的实现文档。
这份文档要足够具体，最好具体到 AI 可以按步骤执行，而不是自由发挥。
实现文档应该包含什么？
至少应该包括：数据库表结构如何设计、是否需要新增字段或迁移数据、后端业务逻辑怎么写、API 如何设计、前端页面如何实现、UI 大致长什么样、状态管理怎么处理、错误提示怎么展示、权限和边界情况如何处理、如何兼容现有逻辑，以及如何拆分实现步骤。
尤其重要的是：实现步骤要拆得足够细。
不要只写"实现用户邀请功能"，而是要拆成：
新增数据库字段；
添加后端接口；
编写业务校验逻辑；
补充单元测试；
实现前端表单；
接入 API；
添加 loading、error、success 状态；
补充 E2E 测试；
回归现有相关功能。这样 AI 在执行时会更稳定，不容易跑偏。
5. 在实现文档中设计测试
测试不能等代码写完后再补。在实现文档阶段，就应该要求 AI 为每一个实现步骤设计对应的测试方案。
每一步都要有测试
每个关键步骤，都应该搭配：单元测试用例、集成测试用例、E2E 测试用例。
至少要覆盖：正常路径、异常路径、权限问题、空数据情况、重复提交、边界值、旧数据兼容，以及回归风险。
这一步同样可以调用 Review Skill，让 Writer 和 Reviewer 结对完成。Writer 负责编写实现文档和测试计划，Reviewer 负责检查遗漏、风险和不合理的设计。
6. 代码实现
到了这一步，才真正开始写代码。
把实现任务交给 Writer，让它严格按照实现文档一步一步执行。
最好分步骤提交
每完成一个独立步骤，最好就 commit 一次。
这样做有几个好处：方便后续 Review、出问题时容易回滚、可以清楚看到每一步改了什么、Reviewer 更容易定位问题，以及 CI 失败时更容易排查。
不要让 AI 一次性改几十个文件，然后再一起提交。那样看起来很快，实际上风险非常高。
更稳妥的做法是：
按实现文档完成一个小步骤；
运行对应测试；
提交 commit；
进入下一步。
7. 验收
代码写完之后，不要急着合并。
这时 AI 应该已经按照实现文档的要求，准备好了单元测试、集成测试和 E2E 测试。
先跑测试
首先跑完整测试集，确保所有测试都通过：单元测试、集成测试、E2E 测试、lint、type check、build。
如果是修复 bug，可以额外要求 AI 在两个分支上分别跑测试：修复前的分支、修复后的分支。这样可以确认：修复前测试确实失败、修复后测试确实通过、测试用例真正覆盖了这个 bug，以及修复不是"碰巧通过"。
这一步对 bugfix 特别重要。
8. 合并
走到这一步，代码已经经历了：上下文分析、PRD 设计、实现文档、测试设计、分步骤实现、AI Review、自动化测试、验收验证。
这时再合并代码，风险会低很多。
如果还是不放心，可以启动应用，手动点一遍核心路径，做一次简单的冒烟测试。但整体来说，只要前面的流程做扎实，合并就不应该再是一次"赌博"。
总结
真正的 vibe coding，不是让 AI 随机发挥。它更像是一套以 AI 为核心的工程流程：
阶段目标关键动作上下文工程让 AI 看懂项目提供项目结构、现有逻辑和业务背景PRD明确要做什么Writer 编写，Reviewer 审查实现文档明确怎么做拆分技术方案和实现步骤测试设计明确如何验证为每一步设计单测和 E2E代码实现按计划执行分步骤实现，分步骤 commit验收降低合并风险跑测试、做回归、验证 bugfix合并交付上线自动化验证通过后再合并
vibe coding 的重点不是"人不看代码"，而是：人把注意力从代码细节，转移到流程设计、上下文质量、验收标准和风险控制上。
当流程足够清楚、上下文足够完整、测试足够严格时，AI 才真的能成为可靠的工程生产力。
