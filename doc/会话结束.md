```md

# Next Action
- <下一步唯一原子动作，必须小到新会话开头可以立刻执行>

- <真实阻塞；没有则 None>

# Key Decisions
- <最多 3 条，只保留影响后续实现的决策>
```

- `Next Action` 必须是原子操作，不能写"完成 auth 模块"这种阶段目标

- Vibe Coding 步骤必须填，便于下次会话快速定位流程位置

## 步骤二：Vibe Coding 状态检查

根据当前所处步骤，决定是否有遗留事项：

| 步骤 | 检查项 |
|---|---|
| `/vibe-feature` 进行中 | Mini-PRD 是否已存入 `doc/features/`？ |
| `/vibe-impl` 进行中 | 实现文档是否已存入 `doc/features/`？ |
| 逐步实现中 | 当前步骤是否已完成？typecheck 是否通过？ |
| 待 `/vibe-accept` | 所有步骤是否已完成？是否已运行验收清单？ |

如有未完成事项，在 `.continue-here.md` 的 `Blockers` 中说明。

## 步骤三：Git 处理

先执行：
```bash
git status --short --branch
npm run typecheck 2>&1 | tail -5
```

分类处理：

| 情况 | 操作 |
|---|---|
| 本轮任务完整、typecheck 通过、已验证 | 精确 commit（功能改动提功能分支，docs/handoff/chore 可直接提 main）|
| 本轮任务未完成 | 不提交、不 stash，通过 `.continue-here.md` 交接 |
| 无关改动 | 跳过，说明原因 |

规则：
- 禁止 `git add .`，只允许精确 `git add <file>`
- 只有 `npm run typecheck` 通过才允许 commit
- 单人开发，不走 PR（详见 AGENTS.md「Git 规范」+「开发流程 Step 4」）：
  - 功能改动：在功能分支（feat/xxx）上 commit + 逐步验收（`/vibe-accept` 全绿）→ 本地 squash merge main → push main 备份
  - docs / handoff / chore 等非功能改动：可直接 commit 到 main
- `push` 失败时保留本地 commit，停止，不重试

提交信息格式：
```
type(scope): 简述本次变更

- Done: <核心完成项>
- Next: <下一步动作>
- Risk: <有则填，无则省略>

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```
