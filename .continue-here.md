# Schema Migration Note（v0.1.9-strategy-composite-pk，2026-07-02）

## v0.1.9 增量（在 v0.1.8 之上）
- schema 从 `v0.1.8-base-ability-rebalance` 升级到 `v0.1.9-strategy-composite-pk`；PRD 仍 v1.0.5。
- **本地 dev DB 必须再次删除重建**（同 v0.1.8 操作流程：`src/main/db/` 无 ALTER 迁移运行器，schema.sql 是 fresh-install 定义）。
- `strategy_config` PK 改动：`strategy_id TEXT PRIMARY KEY`（单列，版本级）→ 表级 `PRIMARY KEY (strategy_id, version)`（复合，族级）。schema 原生支持 PRD「一族一 strategy_id」模型，不再仅靠 UNIQUE(type,job,version) 兜底。
- `assessment_session` / `training_session` 的 FK 改为表级复合 `FOREIGN KEY (strategy_id, strategy_version) REFERENCES strategy_config(strategy_id, version)`；列级 `strategy_id REFERENCES strategy_config(strategy_id)` 删除（列级单列 FK 与表级复合 FK 不能并存）。
- `result_record.strategy_id` 原列级 FK 降级为普通 TEXT（无对应版本列，无法接入复合 FK；该表写入路径仅靠应用层校验，与 PR #1 一致）。
- `UNIQUE(strategy_type, job_code, version)` 保留（DUPLICATE_JOB_STRATEGY 错误码兜底）。
- 触发器无需改动（语义不变：INSERT 时校验 session 引用的 (sid, ver) 存在；UPDATE 时禁止改语义字段；禁 DELETE）。
- handler `createVersion`：复合 PK 下 SQLite 报 PK 冲突为 "UNIQUE constraint failed"（无 "PRIMARY KEY" 字样），无法靠 msg 区分 PK vs UNIQUE(type,job,ver)。改为显式 strategy_id 存在前置检查 + catch 中按 msg 含 `strategy_type` 区分 DUPLICATE_JOB_STRATEGY / DUPLICATE_STRATEGY_ID。
- test-helpers.ts `SCHEMA_PATH` 顺手修：从 `doc/xc-career-guide-mvp-schema-v0.1.7-consistency-guard.sql`（v0.1.7 镜像副本，与生产 schema.sql 漂移）改为 `src/main/db/schema.sql`（单一源，消除漂移）。
- AGENTS.md 工程基线已同步：`schema v0.1.9-strategy-composite-pk | PRD v1.0.5`。

# Schema Migration Note（v0.1.8 / PRD v1.0.5，2026-07-01）
- schema 已从 `v0.1.7-consistency-guard` 升级到 `v0.1.8-base-ability-rebalance`，PRD 从 v1.0.4 升级到 v1.0.5。
- **本地 dev DB 必须删除重建**：`src/main/db/` 无 ALTER 迁移运行器，schema.sql 是纯 fresh-install 定义（`CREATE TABLE IF NOT EXISTS`）。v0.1.8 重命名了 `strategy_config` 列（`pass_threshold`→`competent_threshold` 等），已有的 v0.1.7 本地 SQLite 不会自动迁移，且新 seed INSERT 引用新列名会失败。操作：停止 app → 删除本地 SQLite 文件（通常在用户数据目录下）→ 重启 app 触发 schema.sql 重建。
- `strategy_config` 字段更名：`pass_threshold`→`competent_threshold`(DEFAULT 80)、`improve_threshold`→`conditional_threshold`(DEFAULT 60)；新增表级字段 `module_veto_threshold`(REAL DEFAULT 0.5)、`emotion_collapse_threshold`(INTEGER DEFAULT 3)。
- `scoring_policy_json` 结构变更：不再承载 `competent_threshold / conditional_threshold / module_veto_threshold / emotion_collapse_threshold` 四键（已提升为表级字段）；只保留 `score_values / normalization / safety_override_enabled / level_rules`。**原"双源一致性"校验（json.threshold === 表列 threshold）已废止**，改为 `level_rules` 中 LEVEL_COMPETENT/CONDITIONAL 首条 min 必须等于表列阈值（单一方向：表列为权威源）。
- 等级枚举一次性切换：`LEVEL_PASS/IMPROVE/FAIL` → `LEVEL_COMPETENT/CONDITIONAL/NOT_COMPETENT`（保留 `LEVEL_FAIL_BY_SAFETY`）。`src/shared/types/json-schemas.ts` 已同步。
- seed 两条策略（baseline/mock）统一为 42+8/满分100/80/60，`question_ratio` = `{TRUE_FALSE:14,SINGLE_CHOICE:14,DRAG:14,OFFLINE_OPERATION:8}`，均 CROSS_MODULE 覆盖 6 模块。
- 新增领域事件 `EMOTION_COLLAPSE_THRESHOLD_REACHED`（PRD §4.6/§8.7），payload 定义见 `doc/xc-career-guide-event-payload-schema-v1.0.0.md` §2.9；触发后 `result_record.result_payload_json`(ABILITY_SCORE) 必须落盘 `emotion_collapse_count / module_veto_triggered_by / level_forced_by`（见 JSON 字段规范 §10）。
- 实现文档 `strategy-config-prd.md` / `strategy-config-impl.md` 已同步上述全部变更。
